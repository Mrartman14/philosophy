# Сворачивание крупных карточек маргиналий (clamp + разворот на месте)

Дата: 2026-06-29
Статус: дизайн утверждён, ожидается план

## Проблема

Заякоренные аннотации и комментарии рендерятся карточками на полях документа
(Word-style маргиналии, движок `src/components/anchor-engine`). Высота карточки
напрямую питает stacking-движок: `resolveStack` сдвигает каждую карточку ниже на
высоту предыдущей (`cursor = top + height + gap`, см.
`src/components/anchor-engine/stacking.ts`). Из этого следуют два дефекта при
крупном контенте:

1. **Ломается выравнивание колонки.** Одна большая карточка (длинное тело
   аннотации/коммента ИЛИ длинная цитата фрагмента) сдвигает весь столбец далеко
   вниз и рвёт связь «карточка ↔ якорь», ради которой маргиналия и существует.
2. **Нечитабельность.** В узком поле большой текст превращается в стену.

Цель — ограничить высоту карточки, сохранив возможность прочитать контент
целиком, в стиле комментариев Word/Google Docs (превью + разворот на месте с
переукладкой колонки).

## Решение (обзор)

- **Контент рендерится полностью на сервере** — есть в HTML (no-JS / SEO / a11y),
  ровно как сейчас (карточки — server-компоненты, `AnnotationCard` и превью
  коммента).
- **Клиентское улучшение:** карточки, чьё естественное содержимое превышает
  порог, получают клампнутое превью (`max-height` + нижний фейд-градиент) и
  доступный тоггл «Показать полностью / Свернуть», свёрнутый по умолчанию.
  Карточки в пределах порога — БЕЗ disclosure-обвязки.
- **Разворот на месте.** Любое изменение высоты карточки (разворот/сворачивание,
  догрузка картинки, смена шрифта, async-контент) ловит ОДИН `ResizeObserver` в
  движке и бампит уже существующий `recomputeKey` → `resolveStack` переукладывает
  колонку. Ручной DOM-математики нет; единственный добавляемый замер —
  стандартная проверка переполнения `scrollHeight > порог`.

### Почему не голый `<details>`

Нативный `<details>` — это «всё-или-ничего» (один `<summary>` ИЛИ полный body).
Он не даёт клампнутого превью с фейдом, которое выбрано в дизайне («превью сверх
порога»). Поэтому disclosure делаем как кламп-контейнер + доступный тоггл
(`button` + `aria-expanded` + `aria-controls`), а не bare `<details>`. Исходная
цель «рендер на сервере» при этом сохраняется: контент SSR-ится целиком, кламп —
улучшение поверх; на no-JS виден весь контент без обрезки (корректная деградация,
зеркалит SSR-инвариант `ready=false` в движке).

## Компоненты

### 1. `ClampableContent` (shared client-примитив, новый)

Расположение: shared-слой (`src/components/...`; точное место — на этапе плана,
рядом с UI-kit либо в anchor-engine как презентационный хелпер). Доменно-чистый —
про маргиналию/аннотации/комменты ничего не знает.

- **Пропы:** `maxHeight` (в `rem`), лейблы тоггла (`expandLabel`/`collapseLabel`),
  `children`.
- **Поведение:**
  - Меряет естественную высоту содержимого (`scrollHeight` внутреннего враппера,
    стабилен под `max-height`-клампом, т.к. не зависит от `overflow:hidden`
    родителя).
  - Высота ≤ `maxHeight` → рендерит `children` как есть, **без тоггла и без
    клампа**.
  - Высота > `maxHeight` → оборачивает в кламп (`max-height` + нижний
    masked-фейд) + тоггл; `expanded` — локальный `useState`, по умолчанию
    `false`. Разворот снимает кламп.
  - Пере-замер при изменении размера содержимого (свой `ResizeObserver` на
    внутреннем враппере), чтобы решение «нужен ли кламп» обновлялось при
    async-контенте.
- **a11y:** тоггл — `<button>` с `aria-expanded` и `aria-controls`, указывающим на
  id клампнутой области. Без `tabindex`-хаков, без `role` поверх.
- **Деградация:** до измерения (первый client-рендер / SSR) — `children` целиком,
  без клампа (совпадает с no-JS, без hydration mismatch).

### 2. Движок: `ResizeObserver` → `recomputeKey`

В `MarginNotesColumn` (или выделенном хуке `use-card-resize`) подписываемся
`ResizeObserver`-ом на обёртки карточек (`cardRefs` уже есть). При изменении
`contentRect` высоты — бампим `recomputeKey` (троттл через `requestAnimationFrame`,
чтобы пачка изменений дала один пересчёт). `resolveStack` уже потребляет
`recomputeKey`, так что добавление точечное.

- **Самовосстановление:** заодно чинит латентный баг — сейчас изменение высоты
  карточки по ЛЮБОЙ причине (картинка/шрифт/async) не пересчитывает стек, и
  карточки наезжают.
- **Без петли:** кламп решается по стабильному `scrollHeight` (не зависит от
  применённого клампа); переукладка меняет только `top` соседей, не их высоту →
  их размер не меняется → observer повторно не стреляет.
- **Guard:** `typeof ResizeObserver === "function"` (SSR/старьё) → no-op, движок
  продолжает работать как сейчас (пересчёт по resize/fonts).

### 3. Интеграция в карточки

- `AnnotationCard` (`src/features/annotations/ui/annotation-card.tsx`): тело
  (`AstRender`) оборачивается в `ClampableContent`. Цитата
  (`AnnotationAnchorContext`) — отдельный кламп (актуально на narrow/list; на wide
  цитата и так скрыта через `hideAnchorOnWide`).
- Превью коммента (то, что подаётся в `DocumentCommentLayer` как `preview`) —
  тот же `ClampableContent` вокруг тела.
- Оба домена используют один общий примитив, каркас остаётся за доменом.

## Пороги (дефолты, тюнятся при реализации)

- **Тело:** `max-height ≈ 16rem` (≈ 10–12 строк при `content` `data-size="sm"`).
- **Цитата:** ≈ 3 строки.

Значения — стартовые; финальная настройка по визуальному QA.

## Зоны и риски

- `ClampableContent` — shared (`src/components/...`) → **foundation-уровень**
  (как и сегодняшний движок-фикс). Согласовать размещение на этапе плана.
- `ResizeObserver` — с guard'ом на отсутствие, rAF-троттл против лишних
  пересчётов.
- Кламп через `max-height` + `mask-image`-фейд — логические свойства / RTL-safe
  (нижний фейд по блочной оси, не зависит от направления письма).
- Взаимодействие с подсветкой/выноской: кламп меняет только высоту тела карточки,
  геометрия якоря в тексте не затрагивается; выноска пересчитывается тем же
  `recomputeKey`.

## Тестирование (TDD)

- **`ClampableContent`:**
  - содержимое ≤ порога → нет тоггла, дети целиком;
  - содержимое > порога → есть тоггл, область клампнута;
  - клик по тоггл → `aria-expanded` переключается, кламп снимается;
  - (jsdom: `scrollHeight`/`ResizeObserver` мокаются — прецедент мока RO ниже).
- **Движок:** RO-колбэк при изменении высоты карточки бампит `recomputeKey`
  (мок `ResizeObserver`, проверка повторного пересчёта `resolveStack`).
- **SSR-инвариант:** контент карточки присутствует в DOM при `ready=false` и до
  измерения клампа (нет скрытия за флагом).

## Вне объёма

- Серверное определение «переполнен ли контент» (требует layout — недоступно на
  сервере). Решение клиентское by design.
- Изменение контракта бэка (порядок/размер контента) — не требуется.
- Виртуализация/ленивый рендер длинных списков карточек — отдельная история.
