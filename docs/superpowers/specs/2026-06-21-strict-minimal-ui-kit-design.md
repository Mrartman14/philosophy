# Строгий минимальный UI-kit: оси вариации + границы ответственности

**Дата:** 2026-06-21
**Статус:** дизайн одобрен, готов к написанию плана
**Контекст:** надстройка над инициативой [2026-06-21-ui-kit-base-ui-coverage](2026-06-21-ui-kit-base-ui-coverage-design.md) (вся миграция нативных элементов и base-ui за UI-kit уже выполнена).

## Проблема (корень, не симптом)

Миграция вскрыла повторяющийся класс багов: дефолт обёртки и `className` потребителя целятся в одно CSS-свойство, а наивный `cn` (`src/components/ui/cn.ts`, plain join без tailwind-merge) их не разрешает — исход решает emit-order Tailwind, а не код. Три примера из миграции: Form `flex flex-col gap-4` ломал горизонтальные поисковые формы; IconButton `h-9 w-9` зажимал многострочную строку инбокса; `size-7/8` проигрывали базовым `h-9 w-9`.

**Но `cn`/tailwind-merge — не корень.** Конфликт возникает потому, что на одно свойство есть ДВА источника истины. А двое их потому, что **примитивы залезают в зону ответственности потребителя**: запекают layout/размер (как это поставлено), хотя должны владеть только идентичностью (что это: семантика, a11y, поведение, фокус). `className`-merge как универсальный override — это открытая поверхность, которая по построению создаёт конфликты и позволяет сломать компонент бесконечным числом способов. Дизайн-система должна КОДИРОВАТЬ ограничения; открытый className кодирует обратное.

**Выбранное направление — строгость (закрытые intentful-примитивы), минимум вариантов.** Это сознательный отказ от «открытого kit + tailwind-merge». Цель: kit, который **нельзя применить неправильно**.

## Модель: оси вариации

| Ось | Решение |
|---|---|
| **Размер** | per-component `size` (sm/md/lg) УБИРАЕТСЯ. Один токен высоты контрола; вариация «компактно/просторно» — только через ось плотности (density). |
| **Tone** | 3 семантических тона: `primary` (главное действие, filled) / `neutral` (второстепенное, тихое) / `danger` (разрушительное). `ghost` складывается в `neutral`; иконочный-тихий контрол = `IconButton` (отдельный компонент). Каждый тон — ровно одна каноничная отрисовка. |
| **Layout** | Из поведенческих примитивов УБИРАЕТСЯ. Один layout-примитив `Stack` (вертикальный ритм, gap от density). |
| **className (core-стиль)** | Leaf-контролы НЕ принимают `className` для вида/размера/фона — только типизированные пропы. |

### Размер через плотность, скоупимую на поддерево

Токены уже каскадируют: `--size-control-h-md` (и pad-токены) переопределяются в `[data-density="compact"]` (`src/styles/tokens.generated.css`). Значит ось плотности — это единственная ось размера, управляемая:
- **глобально** — пользовательская настройка appearance (уже есть);
- **локально** — любой плотный регион (тулбар редактора, компактные тогглы) оборачивается в `<div data-density="compact">`, и все контролы внутри сжимаются через токен-каскад, БЕЗ единого per-component пропа.

Компоненты привязываются к ОДНОМУ control-height токену (`--size-control-h-md` как «высота контрола»); тиры sm/lg из публичного API компонентов уходят (в CSS остаются только если что-то структурное реально требует — иначе удаляются). `IconButton` переводится с литерала `h-9 w-9` на тот же токен (квадрат высотой контрола) — становится density-aware, как `Button`.

## Граница: leaf vs structural

Это ключ к className-политике.

- **Leaf-контролы — ЗАКРЫТЫ** (только пропы, ноль className-для-вида): `Button`, `IconButton`, `Select`, `TextInput`, `Textarea`, `Checkbox`, `Label`. Ширина/позиция — забота родителя: по умолчанию контрол занимает ширину контейнера; нужна узкая (фильтр) — родитель сужает (обёртка/Stack-ширина), а не контрол через `w-auto`.
- **Structural / compound — ОТКРЫТЫ** (композиционные поверхности, className = раскладка): `Stack`, `Toolbar`, `Popover`, `NavigationMenu`, `Dialog`, `Table`. Здесь className легитимен и не конфликтует с «видом контрола».
- **Form** — поведение only (`errors`/`onSubmit`/`action`/`ref`), без layout-className.

`unstyled` Button остаётся как ЕДИНСТВЕННЫЙ явный escape (именованный проп, не className) — для кликабельных строк/карточек.

## Минимальный API (целевой)

```tsx
// Leaf-контролы: пропы, без className-для-вида
<Button tone="primary" type="submit">Сохранить</Button>     // default tone = neutral
<Button tone="neutral" onClick={onCancel}>Отмена</Button>
<IconButton tone="neutral" aria-label="Удалить"><TrashIcon/></IconButton>
<Select name="visibility" options={[…]} aria-label="Видимость" />

// Поведение + раскладка композицией
<Form action={action} errors={errs}>
  <Stack>
    <FormField name="title" label="Заголовок" required>
      <TextInput name="title" required />
    </FormField>
    <Select name="visibility" options={[…]} aria-label="Видимость" />
  </Stack>
</Form>

// Плотный регион — через контейнер, не пропы
<div data-density="compact">
  <Toolbar.Root>{/* кнопки сжаты токен-каскадом */}</Toolbar.Root>
</div>
```

### `Stack` (новый примитив)
Вертикальный ритм. Минимальный API: рендерит `div` с `flex flex-col`, gap из density-токена (`--space-stack-gap` или переиспользование `--space-control-pad`-семейства — уточняется планом). Возможен один проп `gap?: "sm"|"md"` ТОЛЬКО если доказан спрос; по умолчанию — без пропов (один канонический gap). Это structural-примитив → className разрешён (для редких раскладочных отличий).

## Enforcement (TS + ESLint)

- **TS-уровень (строже линта):** leaf-контролы НЕ объявляют `className` в своих Props → передача = ошибка компиляции. `size` убран из типов. `tone` — узкий union из 3 значений.
- **Guardrail 8 (ESLint, по образцу Guardrail 7):** запрет устаревших variant-имён (`variant="secondary|ghost"`) и `size=` на kit-контролах; точечный запрет сырого `flex flex-col`/`grid`-layout на форм-уровне вне `Stack`/structural (узко, чтобы не зашумлять).
- **Guardrail 7** (ноль нативных тегов / прямых base-ui вне kit) — остаётся.

## Фазы и ЧЕСТНАЯ цена

Это надстройка над свежей миграцией и ПОТРЕБУЕТ повторного касания ~40 потребителей, которые сейчас передают `className`/`size` в leaf-контролы. Фазы (каждая — отдельный шиппабельный срез):

1. **Токен-консолидация размера.** Один control-height токен; `IconButton` на токен; подтвердить density-скоуп через `data-density` на контейнере. Убрать только что добавленный `IconButton.size` (Task 15 предыдущей инициативы) — заменить плотным регионом у его 3 потребителей.
2. **`Stack`-примитив** + тест.
3. **Tone-консолидация.** `Button`/`IconButton`: 3 тона, `ghost`/`secondary`→маппинг, default `neutral`. Ре-миграция call-site'ов.
4. **Закрытие className на leaf-контролах** (TS). По одному компоненту: убрать `className` из Props, перенести редкие позиционные нужды на родителя/`Stack`/явный проп; ре-миграция потребителей этого компонента.
5. **Form без layout** → потребители оборачивают поля в `Stack` (снять Form-дефолт `flex flex-col gap-4`, добавленный в прошлой инициативе).
6. **Guardrail 8** (последним — чтобы lint зеленел только после миграции).

## Риски и решения, принятые за пользователя (одобрены)

- **default tone = `neutral`** (громкость opt-in; `primary` — явный выбор для главного действия). Меняет много call-site'ов, но делает иерархию явной.
- **Leaf-контролы полностью без className** (даже позиционного). Если где-то нужна позиционная привязка — добавляется явный проп ИЛИ родитель/`Stack`, не className. Самое строгое; основной источник ре-миграции.
- **Плотность скоупится через `data-density` на контейнере**, не новым пропом — переиспользует существующий токен-каскад.
- **Stack-ширина для узких контролов:** «фильтр должен быть узким» решается шириной контейнера/Stack, не `w-auto` на Select. План должен дать каноничный приём (например `Stack` или обёртка с `max-w`).
- **Совместимость с base-ui:** base-ui-части (Select trigger, Checkbox) принимают `className` через render/часть — закрытие касается ВНЕШНЕГО API kit-обёртки, не внутренней композиции base-ui.
- **Объём:** это сравнимо по объёму с предыдущей миграцией; делается строго пофазно, каждая фаза зелёная по `pnpm lint && pnpm test && pnpm build` (с учётом, что pre-existing долг semantic-map/canvas — не наш).
