# Выноски-связи маргиналий ↔ текст (leader lines)

Дата: 2026-06-26
Статус: дизайн одобрен, ожидает плана

## Проблема

На странице лекции/документа маргиналии живут в боковых колонках: комментарии — в
левом поле, аннотации — в правом. Карточка стоит примерно напротив текста-якоря, но
алгоритм раздвижки `resolveStack` ([stacking.ts](../../../src/components/anchor-engine/stacking.ts))
сдвигает пересекающиеся карточки вниз. Из-за этого карточка часто стоит **не**
напротив своего фрагмента, и связь «нода текста ↔ карточка» теряется.

Нужна ясная визуальная связь карточки с её фрагментом — по образцу выносок Word —
без перечёркивания текста линиями и без оверинжиниринга.

## Решения (из брейншторма)

- **Видимость:** связи видны **всегда**, для **всех** карточек, на **обеих** сторонах.
- **Смена политики комментариев:** левая сторона (комментарии) переходит из текущего
  «тихого» lazy-режима (`InlineAnchorLayer`: подсветка по умолчанию выключена,
  карточка по hover) в **eager** — постоянная подсветка + карточки + выноски, как у
  аннотаций. Это осознанное изменение поведения.
- **Форма линии:** ортогональный «локоть» в стиле Word — короткий горизонтальный ус
  от края текста → вертикаль в жёлобе → короткий ус к карточке.
- **Цвет:** два типа-цвета. Комментарии — один оттенок, аннотации — другой. Один и тот
  же цвет красит линию + подсветку текста + акцент карточки.
- **Hover-акцент:** все линии видны приглушённо; при наведении на карточку **или** на
  текст-якорь её линия + подсветка становятся ярче/толще, остальные слегка гаснут.

## Геометрические инварианты (почему текст не перечёркивается)

1. Линия стартует **ровно на внутренней границе текстовой колонки** (не от глифа, а от
   края колонки на высоте якоря) и далее лежит **только в жёлобе/поле**. По горизонтали
   все точки пути находятся в диапазоне `[край_текстовой_колонки … край_карточки]` —
   в зону глифов путь не заходит **по построению**.
2. Сторона (лево/право) выводится **из геометрии** (карточка левее или правее текста),
   а не из логического `side` — это делает решение RTL-safe автоматически.
3. `resolveStack` сортирует карточки по позиции якоря и раздвигает их монотонно вниз,
   сохраняя **порядок** карточек = порядку якорей. Следствие: выноски монотонны и
   **между собой не пересекаются** — классического «спагетти» не возникает.

## Архитектура

Один чистый модуль геометрии + один тонкий SVG-слой + минимальная обвязка поверх
существующего движка маргиналий. Ничего в запретных зонах (`src/api/schema.ts`,
shell-файлы, UI-kit, общая инфраструктура `src/utils|hooks|services`).

### Компоненты

1. **`src/components/anchor-engine/connector-geometry.ts`** — чистая функция.
   `connectorPath({ anchorY, textEdgeX, cardY, cardEdgeX, side, stub }) → string`
   возвращает SVG-атрибут `d` для ортогонального локтя. Без DOM, полностью
   юнит-тестируема.

2. **`src/components/anchor-engine/connector-layer.tsx`** — **один** SVG-оверлей на
   страницу. Охватывает весь грид-контент: `position:absolute; inset:0;
   pointer-events:none; aria-hidden`. Координаты считаются **относительно грид-обёртки**
   (той же скролл-обёртки, в которой позиционируются карточки), поэтому линии
   «приклеены» к контенту при скролле **без** scroll-листенеров. Рендерит `<path>` на
   каждую видимую заметку; CSS-класс цвета по типу (comment/annotation) + классы
   `is-active` / `is-dimmed` по `activeNoteId`.

3. **Сбор координат** — в существующем `useLayoutEffect` из
   [margin-notes-column.tsx](../../../src/components/anchor-engine/margin-notes-column.tsx),
   где уже измеряются якоря и резолвится стек. Добавляем сбор `anchorY` (вертикальный
   центр первой строки якоря) и `cardY` (резолвнутый top карточки), переводим в
   грид-координаты и отдаём в `ConnectorLayer`.

4. **`activeNoteId`** — лёгкий общий стейт (наведение на `[data-note-card]` или на
   текст-якорь через существующий hit-test), поднят в
   [margin-anchor-layer.tsx](../../../src/components/anchor-engine/margin-anchor-layer.tsx).
   Потребляется слоем линий **и** контроллером подсветки. Акцент реализуется чисто
   CSS-классами (без перерисовки геометрии).

5. **Унификация комментариев** — сторона комментариев маршрутизируется через тот же
   eager-путь, что и аннотации (как в `MarginAnchorLayer`), параметризованный
   `side="start"` + цвет комментария + источник данных комментариев. Это самый крупный
   код-тач: [document-comment-layer.tsx](../../../src/features/comments/ui/document-comment-layer.tsx)
   уходит с lazy `InlineAnchorLayer` на eager-рендер. Доступный список заметок
   (клавиатура/AT) сохраняется.

6. **CSS** — два цвет-токена (comment / annotation) для линии + подсветки + акцента
   карточки; классы emphasis/dim; видимость выносок **только на xl+** (где включено
   абсолютное позиционирование карточек).

### Поток данных

```
TextAnchor
  → useAnchorRanges → Range → getBoundingClientRect  ⟶ anchorY, textEdgeX
  → resolveStack                                      ⟶ cardY
  → DOM-рект карточки                                 ⟶ cardEdgeX
  → connectorPath(...)                                ⟶ SVG `d`
  → <path>
activeNoteId (hover карточки/текста) → CSS-акцент (is-active / is-dimmed)
```

Пересчёт геометрии — в том же layout-цикле, что и позиционирование карточек
(resize / смена контента). Отдельных scroll-листенеров нет.

### Форма локтя (right-side, карточка правее текста)

```
A = (textEdgeX, anchorY)            // внутренняя граница текстовой колонки, на высоте якоря
P1 = (textEdgeX + stub, anchorY)    // короткий горизонтальный ус в жёлоб
P2 = (textEdgeX + stub, cardY)      // вертикаль в жёлобе
D = (cardEdgeX, cardY)              // короткий ус к внутреннему краю карточки
```

Для left-side (карточка левее текста) — зеркально. Сторона выводится из сравнения
`cardEdgeX` и `textEdgeX`, не из логического `side`.

## Граничные случаи и обработка

- **Узкие экраны (< xl):** поля схлопываются, карточки втекают в хребет (`margin-note--inline`).
  Выноски **не рендерятся** (иначе пересекли бы текст). Тот же брейкпоинт, что у
  абсолютного позиционирования карточек.
- **Якорь не резолвится** (дрейф / блок удалён): линия для этой заметки **пропускается**
  (guard на `null` из `rangeFromAnchor`). Карточка остаётся в доступном списке. Без throw.
- **Карточка ещё не измерена:** пропуск до следующего layout-прохода.
- **reduced-motion:** при `motion=reduced` — без анимированных переходов акцента
  (мгновенная смена состояния).
- **a11y:** линии декоративны (`aria-hidden`, `pointer-events:none`). Клавиатурный/AT-путь —
  существующий доступный список заметок, не меняется.
- **Перф:** комментарии теперь eager — потенциально больше карточек и линий. Машинерия
  та же, что у аннотаций (уже eager). Помечено как watch-item, не блокер.

## Тестирование

- **Чистая геометрия (TDD):** инварианты — все x пути в `[textEdgeX … cardEdgeX]` и
  никогда в зоне текста; сторона выводится из геометрии; точки локтя корректны для
  обеих сторон.
- **Монотонность:** свойство «порядок карточек сохранён ⇒ линии не пересекаются».
- **Компонент:** одна `<path>` на видимую заметку; `activeNoteId` акцентирует/гасит;
  на узком экране линий нет; нерезолвящийся якорь пропускается.
- **Регрессия:** обновить тесты комментариев под eager-поведение
  (вместо lazy hover-reveal).

## Бэкенд

Фича чисто фронтовая. Контракт anchor (UTF-16, `target_entity_*`) уже достаточен —
новых требований к бэку нет. Если по ходу всплывёт дефект контракта anchor — флагуем
пользователю по правилу FE/BE-разделения, а не маскируем.

## Не входит в объём (YAGNI)

- Цвет по автору / по треду.
- Кривые/безье и диагональные линии (выбран ортогональный локоть).
- Выноски на узких экранах.
- Read-only рендер вне страниц лекции/документа.
