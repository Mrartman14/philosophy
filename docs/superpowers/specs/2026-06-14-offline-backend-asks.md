# Бэкенд-бриф (philosophy-api): foundation под офлайн-режим фронта

**Дата:** 2026-06-14
**Кому:** команда `philosophy-api` (Go, stdlib `net/http`)
**Зачем:** фронт строит **generic offline foundation** — переиспользуемую базу для (а) офлайн-кеша сущностей для чтения и (б) офлайн-создания с синком. Нужны три бэкенд-возможности; **все три — generic/переиспействуемые** (middleware + composing-пакет + реестр), т.е. вложение один раз под любые сущности, а не точечные заплатки.

Фронт **не блокируется**: стартуем на фронтовой оркестрации существующих GET + single-drain очереди; рычаги подключаем по мере готовности. Порядок ценности: **1 → 2 → 3**.

> ✅ **Вопрос про статику закрыт** (ваш ответ): при `StorageType=local` `/static/files/` раздаётся тем же процессом на том же origin (`mux.Handle("/static/files/", …)`), CORS — allowlist `CORS_ORIGINS`. Same-origin подтверждён → клиентский `fetch` картинок в Cache Storage работает без отдельного CORS. Если когда-нибудь S3/CDN — понадобится CORS на бакет (не в текущих планах). Учтём как риск, доработки сейчас не нужно.

## Рычаг 1 — Generic offline-bundle endpoint `GET /api/{entity}/{id}/bundle`

**Проблема:** офлайн-снимок сущности фронт сейчас собирает из множества GET (лекция + теги + все страницы комментов + аннотации). Это N round-trips и плохо обобщается (упирается в границы фич на фронте).

**Что просим:** один эндпоинт, отдающий структурный JSON-снимок: корневая сущность + связанные данные + version-токен, **собранный сервером**.

- Ложится в ваш паттерн **composing-only пакета** — прецедент `internal/llmsindex/` (`service.go`+`handler.go` без repo, оркестрирует несколько сервисов). Новый `internal/bundle/` делает fan-in уже существующих сервисов.
- Половина данных уже агрегируется lecture-scoped роутами (`…/comments`, `…/annotations` через `annotationHandler.ListByLecture`, `…/documents`, `…/media`, `…/tags`) — bundle их переиспользует; `optionalAuth` + `shareTokenMW` наследуются (visibility/доступ бесплатно).
- Снимает N+1 на комменты: сервер сам прокручивает страницы внутри транзакции (внутренняя агрегация может игнорировать публичный `max limit=100`).
- **Обобщение:** реализовать через реестр `map[entityType]BundleAssembler` (как у вас `annotationParentCheckers`/`annotationBlockValidators` в `main.go`). Начать с `lecture` (наибольший выигрыш), сигнатуру сразу сделать generic.

**Форма ответа (предложение):** ваш стандартный envelope `{data: {...}}`, где `data` = `{ <entity>: <root>, <relatedKey>: [...], ... , version: <token> }`. Точную форму согласуем — фронту важно: один вызов, со всеми связанными, с version-токеном (см. рычаг 2).

**Приёмка:** `GET /api/lectures/{id}/bundle` отдаёт лекцию + теги + комменты (все) + аннотации одним ответом; уважает visibility и `?token=` так же, как существующие роуты.

## Рычаг 2 — ETag / `If-None-Match` на JSON GET + bundle

**Проблема:** фронту нужно дёшево понять «протух ли сохранённый офлайн-снимок», не перекачивая его. И — это же даёт version-токен для будущих офлайн-**апдейтов** (оптимистичная конкуренция).

**Что просим:**
1. На `…/bundle` (и желательно на обычных JSON `GET /api/{entity}/{id}`) возвращать `ETag` и поддерживать `If-None-Match` → **304 Not Modified**.
2. Для bundle ETag — составной (например, хэш от набора `updated_at` входящих сущностей или max `updated_at`).
3. Добавить `updated_at` сущностям, где сейчас только `created_at`: **media, tag, sharelink** — иначе их staleness нечем определить.

- Инфра уже есть: `WriteRenderedForRequest` (`internal/httputil`) делает sha256-ETag + 304 для `.md`/`.txt`; конвенция `docs/conventions/optimistic-locking.md` уже канонизировала формат ETag (raw `updated_at`, TEXT-колонка, timestamp с Go-стороны); canvas уже отдаёт ETag для JSON. Нужно распространить на bundle/JSON-GET.
- ⚠️ Ловушка из вашей же конвенции: byte-stable round-trip `updated_at` в `modernc.org/sqlite` (обрезка trailing zeros) — учесть при включении ETag на новых сущностях.

**Приёмка:** повторный `GET …/bundle` с `If-None-Match: <prev-etag>` → `304` без тела, если ничего не менялось; иначе `200` + новый ETag.

## Рычаг 3 — Idempotency-Key middleware (для офлайн-создания)

**Проблема:** офлайн-созданные сущности переотправляются при возврате сети. Ретрай семантически **at-least-once**: ответ на успешный POST может потеряться (обрыв уже ПОСЛЕ коммита), клиент, не получив 2xx, шлёт повторно → **дубликат**. Клиент не отличает «запрос не дошёл» от «ответ не дошёл». Дедуп обязан быть на сервере.

**Главное требование — платформенный middleware, НЕ логика в хендлере аннотаций.** Generic, по заголовку `Idempotency-Key`, подключаемый к любому write-роуту декларативно. Цель — будущие офлайн-записи (комменты, реакции, …) подключаются без правок middleware.

- Чистое место — глобальная цепочка `cmd/server/main.go:~1096` (рядом с `RequestID`); `internal/middleware/requestid.go` — готовый шаблон. Либо per-route обёртка для POST-семейства.

**Спецификация:**
- Заголовок `Idempotency-Key` (клиентский UUID v4). Опционален; без него — поведение как сейчас.
- Запись дедупа: `(user_id, key) → { request_fingerprint = hash(method+path+body), response_status, resource_id | response_body, created_at }`. Скоуп `(user_id, key)`; fingerprint защищает от переиспользования ключа на другой ручке/теле.
- **Первый** запрос: выполнить, сохранить.
- **Повтор (тот же ключ + тот же fingerprint):** НЕ выполнять; вернуть **тот же результат — 2xx + то же тело (тот же `id`)**. ⚠️ Критично: реплей обязан быть **2xx** (фронт-синк трактует не-2xx как ошибку).
- **Тот же ключ + другой fingerprint:** `422 IDEMPOTENCY_KEY_REUSED`.
- **Конкурентный in-flight:** сериализовать и отдать результат, либо `409 IDEMPOTENCY_KEY_IN_USE`.
- TTL записи: ≥ **7 дней** (офлайн-синк может задержаться).
- Скоуп ключа — `user_id` из `rbac.FromContext`.

**Применяем сейчас:** create аннотаций (`POST /api/{documents|comments|glossary|media}/{id}/annotations`). Механизм — generic, готов к расширению.

**OpenAPI:** задокументировать опциональный header `Idempotency-Key` на этих роутах (фронт генерит типы из swagger). Схема ответа не меняется.

**Приёмка:** повтор create с тем же ключом+телом → 2xx + та же сущность (тот же id), в БД 1 объект (не 2); без заголовка — как раньше; разные юзеры с одним ключом — независимы; другой fingerprint → 422; подключение middleware к новому write-роуту не требует правок самого middleware.

## Порядок и связь с фронтом

1. **bundle** (рычаг 1) — фронт переключит `assemble` с оркестрации на bundle-endpoint бесшовно (дескриптор прячет источник).
2. **ETag** (рычаг 2) — фронт начнёт слать `If-None-Match` для проверки протухания + откроет дорогу офлайн-апдейтам (уровень 2).
3. **idempotency** (рычаг 3) — фронт включит безопасные ретраи синка (сейчас риск дублей снижен single-drain, но не убран).

Все три — generic; их отсутствие фронт переживает (деградация), наличие — делает решение чистым.
