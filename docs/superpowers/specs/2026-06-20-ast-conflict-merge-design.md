# Дизайн: разрешение конфликта версий AST-контента (поблочный merge)

Дата: 2026-06-20

## Проблема

Бэкенд использует оптимистическую блокировку: на сохранение устаревшей версии
сущности возвращает `412 VERSION_MISMATCH`. Фронт шлёт `If-Match: "<version>"` из
скрытого поля формы — паттерн единообразен для всех версионируемых сущностей
(`document`, `annotation`, `comment`, `event`, `glossary`, `banner`, `trail`),
см. [src/utils/optimistic-lock.ts](../../../src/utils/optimistic-lock.ts).

Текущий UX при конфликте примитивен: показываем локализованный текст «Объект
изменён в другом месте. Обновите страницу и повторите» (см.
[src/i18n/messages/ru/errors.ts](../../../src/i18n/messages/ru/errors.ts)). Юзер
обновляет страницу — **и его несохранённые правки теряются**. Когда два человека
одновременно правят один документ, проигравший гонку теряет работу и не видит, что
именно изменил другой.

## Цель

Дать пользователю, попавшему в конфликт версий, **поблочно объединить** свои правки
со свежей серверной версией, ничего не теряя и видя, что изменил другой человек.
Механизм **entity-agnostic**: работает для любой сущности, редактирующей AST-блоки
через общий редактор ([src/components/ast-editor/](../../../src/components/ast-editor/)).
Документ — частный случай.

## Ключевые решения (зафиксированы в брейншторме)

1. **Ядро — AST-контент, не документ.** Механизм оперирует `AstBlock[]`
   ([src/components/ast-editor/types.ts](../../../src/components/ast-editor/types.ts)),
   не доменными полями. Документ — частный случай.
2. **Мощность разрешения — поблочный merge** (не бинарный «оставить/перезаписать»
   и не git-подобный авто-merge): пользователь решает по каждому конфликтному блоку.
3. **Base-aware классификация с полной видимостью.** Снимаем `base` (версия на момент
   открытия формы), чтобы различать «конфликт» (правили оба) и «одностороннее
   изменение». Но серверные изменения **не прячем** — показываем как контекст, потому
   что они могут менять смысл правок пользователя.
4. **Раскладка — единый результат + inline-конфликты.** Одна колонка = будущий
   документ, читается в потоке; конфликты раскрываются прямо в тексте.
5. **Поток разрешения — выбор inline → финальный проход в редакторе.** После выбора
   по конфликтам собранный результат грузится обратно в обычный `LazyAstEditor` для
   свободной доводки (сшить швы, совместить формулировки) перед сохранением.
6. **Поверхность — полноэкранный takeover** поверх формы. Закрыл → вернулся к форме.
7. **Источник свежей версии — FE-refetch сейчас, контент в 412 потом.** Дизайн не
   блокируется на бэкенде (см. §«Источники данных»).
8. **Внутриблочный word-diff (текстовый) в MVP** для конфликтных и `server-only`
   блоков. Структурный mark-aware diff — отложен.
9. **Первая волна — только blocks-only сущности.** Скалярные поля откладываются
   (см. §«Скалярные поля и границы первой волны»).

## Инвариант, на котором всё держится: стабильный `block.id`

Проверено по коду: `ast.Block.id` (`string`) присваивается **бэкендом** и уникален
в пределах документа; переживает round-trip сохранения (
[src/components/ast-editor/serializer.ts](../../../src/components/ast-editor/serializer.ts),
[deserializer.ts](../../../src/components/ast-editor/deserializer.ts),
инвариант зафиксирован в
[dedup-block-id-plugin.ts](../../../src/components/ast-editor/extensions/dedup-block-id-plugin.ts)).

Следствия для diff:
- Блоки с одинаковым `id` в разных версиях — **один и тот же блок**.
- Новый/несохранённый блок имеет пустой `id` (`""`) → всегда «добавлен мной».
- Порядок = позиция в массиве (`position` = индекс), отдельного поля сортировки нет.

## Сквозной поток

```
1. Юзер правит блоки, Save → PUT с If-Match: "<base_version>"
2. Бэкенд: 412 VERSION_MISMATCH
3. FE достаёт theirs (свежие blocks + version):
     - есть контент в detail 412 (после бек-фикса) → берём оттуда
     - иначе → fetchFresh(id) через read-endpoint сущности
4. classifyBlocks(base, mine, theirs) → merge-модель
5. Полноэкранный takeover: единый результат + inline-конфликты + word-diff
6. Юзер решает конфликты (сервер/моё), видит server-only изменения как контекст
7. «Применить» → собранные блоки грузятся обратно в LazyAstEditor
8. Финальная доводка руками → Save с If-Match: "<theirs_version>"
9. Если снова 412 → возврат к шагу 3 с новым theirs
```

`base` доступен всегда — это блоки, отрендеренные в форму при загрузке (снимок).
`mine` — текущее состояние редактора в памяти; на 412 редактор остаётся смонтирован,
правки не теряются.

## Ядро diff/merge (чистая функция, TDD)

`classifyBlocks(base, mine, theirs): MergeEntry[]` — чистая, без React,
исчерпывающе юнит-тестируемая. Сопоставление по `block.id`; пустой `id` =
локально добавленный блок (сопоставляется по позиции в редакторе).

Статусы блока (`MergeEntry.status`):

| Статус | Условие (относительно `base`) | Поведение по умолчанию | В UI |
|---|---|---|---|
| `unchanged` | не менялся ни у кого | берём как есть | свёрнут |
| `mine-only` | менял только я | беру моё | бейдж «ваша правка», виден |
| `server-only` | менял только сервер | беру серверное | бейдж «изменено на сервере», виден, word-diff |
| `conflict` | меняли оба | **не выставлен — спрашиваем** | раскрыт, два варианта + word-diff |
| `added-mine` | новый блок у меня | беру | бейдж «добавлен вами» |
| `added-server` | новый блок на сервере | беру | бейдж «добавлен на сервере» |
| `removed-mine` | я удалил, сервер не менял | беру удаление | бейдж «удалён вами» |
| `removed-server` | сервер удалил, я не менял | беру удаление | бейдж «удалён на сервере» |
| `structural-conflict` | один удалил, другой изменил | **спрашиваем** (оставить правку / принять удаление) | раскрыт |

**Правило порядка (зафиксировано):** спина результата = порядок `theirs` (последняя
зафиксированная реальность); блоки `added-mine` вставляются после своего якорного
предшественника. Сложные перестановки детерминированно не разрешаются — это
осознанно: финальный проход в редакторе (шаг 7) — клапан безопасности для ручной
доводки порядка и швов.

## Внутриблочный word-diff (текстовый)

Для `conflict` и `server-only` блоков подсвечиваем добавленные/удалённые **слова**
(VSCode-подобно): `theirs` vs `base` = «что изменил сервер», `mine` vs `base` =
«что изменил я». Дифф по **тексту** блока (`block.text` / извлечённый плоский текст);
разметка (marks) при сравнении игнорируется, но рендерится. Структурный mark-aware
diff (подсветка «стало жирным/ссылкой») — отдельная будущая итерация.

Реализация — отдельная чистая функция `wordDiff(baseText, sideText): DiffToken[]`,
юнит-тестируемая независимо от React.

## Merge-UI (полноэкранный takeover)

Компонент `<AstMergeView>` — entity-agnostic. Вход: `{ base, mine, theirs }`. Выход:
собранные `AstBlock[]` (через `onApply`).

- Единая колонка = будущий документ, читается в потоке.
- Блоки рендерятся read-only (рендер AST), помечены бейджами по статусу.
- `conflict` / `structural-conflict` раскрыты inline: два варианта (сервер / моё) с
  радио-выбором и word-diff. Кнопка «Применить» активна, только когда все
  конфликты решены.
- `unchanged` свёрнуты в «N блоков без изменений», разворачиваются по клику.
- «Применить и продолжить» → собирает блоки по выборам и порядковому правилу,
  закрывает takeover, пишет результат в `LazyAstEditor`.

## Источники данных, деградация и бек-ask

Адаптер на сущность (в духе descriptor-паттерна офлайна):
`fetchFresh(id) → { blocks, version }`, **server-only**. Пока бэкенд не кладёт
контент в тело 412 — используется он. Когда положит — читаем из `detail`, тот же UI,
адаптер становится фолбэком.

**Бек-ask (передан пользователю для бэкенда):** класть текущие `blocks` + `version`
конфликтной сущности в тело ответа `412 VERSION_MISMATCH` (например в `detail`) для
всех версионируемых сущностей с блоками. Тогда конфликт разрешается одним
раунд-трипом без гонки повторного чтения. **Это не блокер** — дизайн работает и без
него.

Деградация: если у сущности нет read-пути (`comment` — нет single-GET) и бэк ещё не
отдаёт контент в 412 → для неё временно остаётся текущий примитив «обновите
страницу». Раскатка merge по сущностям инкрементальная, по готовности источника.

## Скалярные поля и границы первой волны

«Скалярные» = не-блочные поля сущности (`title`, даты, флаги). Они тоже могут
разойтись при параллельной правке. Ключевое различие — **endpoint сущности**:

| Сущность | Endpoint | Что расходится при 412 |
|---|---|---|
| `document` | `PUT /documents/{id}/blocks` | только блоки |
| `glossary`, `comment`, `annotation` | blocks-only / blocks-доминантный | в основном блоки |
| `event`, `banner` | полный body | блоки **и/или** скаляры |

Для полнопейлоадных сущностей 412 может прилететь из-за расхождения только скаляра
(блоки идентичны). Тогда поблочный merge покажет «конфликтов нет», юзер сохранит со
свежей версией — и **молча затрёт серверный скаляр**. Это тихая потеря.

Поэтому **первая волна merge = только blocks-only сущности** (документ во главе), где
скаляры физически не могут разойтись. Полнопейлоадные (`event`, `banner`) подключаются
во вторую волну вместе с мини-механизмом скаляров («поле изменилось: сервер=X,
ваше=Y — выбрать») — отдельный UI-паттерн, требующий знания доменных полей, поэтому
**вне** entity-agnostic ядра. До второй волны у них остаётся текущий примитив, без
риска тихой потери.

## Generic-интеграция

- Чистое ядро `classifyBlocks` + `wordDiff` + типы `MergeEntry`/`DiffToken` — в общем
  месте (предв. `src/components/ast-merge/`; финальное размещение — по конвенциям
  слайсов и запретных зон из [AGENTS.md](../../../AGENTS.md)).
- `<AstMergeView>` — entity-agnostic UI-компонент.
- Клиентский оркестратор поверх существующей формы блоков: ловит
  `code === "VERSION_MISMATCH"`, дёргает `fetchFresh` (или читает `detail`), открывает
  takeover, по `onApply` пишет результат в редактор.
- Форма блоков получает два новых пропа: `baseBlocks`/`baseVersion` (снимок загрузки) и
  `fetchFresh`-адаптер. Прочие edit-формы подключаются однотипно по мере раскатки.
- Контракт `If-Match` и server actions **не меняются** — конфликт по-прежнему
  возвращает `412 VERSION_MISMATCH`; меняется только клиентская обработка этого кода.

## Краевые случаи

- **Повторный 412** при сохранении merge (кто-то успел сохранить) → возврат к шагу 3
  с новым `theirs`.
- **Сущность удалена на сервере** (404 при `fetchFresh`) → сообщение «удалено в
  другом месте», выбор: сохранить как есть (заново создать) / отбросить.
- **Мои несохранённые блоки** (`id = ""`) → всегда `added-mine`, сопоставление по
  позиции в редакторе.
- **Скаляры разошлись** на blocks-only сущности — невозможно (endpoint не несёт
  скаляров); на полнопейлоадных — вне первой волны (см. выше).

## Тестирование

- `classifyBlocks` — обширные юнит-тесты на все статусы, правило порядка, пустые `id`,
  `structural-conflict` (TDD, чистая функция).
- `wordDiff` — юнит-тесты на вставку/удаление/перемешивание слов, пустые строки.
- Оркестратор: `412 → fetchFresh → takeover`; повторный `412 → новый раунд`;
  `404 → ветка удаления`.
- UI `<AstMergeView>`: бейджи по статусам, блокировка «Применить» до решения всех
  конфликтов, корректная сборка результата и запись в редактор.
- Регрессия: существующие optlock-тесты остаются зелёными (контракт `If-Match` не
  меняем).

## Что вне охвата

- Структурный mark-aware word-diff.
- Скалярные поля и полнопейлоадные сущности (`event`, `banner`) — вторая волна.
- Real-time коллаборация (CRDT/OT) — другая архитектура, не рассматривается.
- Проактивное предупреждение «документ изменился» во время редактирования (до Save).
